# AUTH.md — the auth rail (canon)

A racecar surface exposes `api` to the outside. The outside is untrusted. So **every generated
surface is closed by default**: it refuses a request that does not carry a valid credential AND the
scope the requested command requires. Auth is not an add-on a deployer remembers; it is canon, the
identity analog of the write rail, generated by default and enforced mechanically. This file is the
framework-neutral doctrine; the Django implementation is the `racecar-secure-server` skill (the
Authorization Server) and [`GENERATION.md`](GENERATION.md) (the resource-surface rail).

## The model: one auth path, both surfaces

`lib -> api -> surfaces`. Auth sits on the surface boundary, the same on every surface:

- **One credential, one path.** REST and MCP expose the same command set (both derive from one
  binding), so they authenticate identically: an **OAuth 2.1 opaque bearer token**. No per-surface
  divergence, no second scheme. A surface is an OAuth 2.1 **resource server**.
- **Opaque tokens, never JWT.** The token is a random, server-side handle carrying no claims on the
  wire. It is short-lived, scoped, and revocable. The resource surface validates it by **introspection**
  (RFC 7662) against the Authorization Server, cached briefly; it never trusts a self-describing token.
- **A separate Authorization Server** is the only stateful piece. Humans authenticate there with
  **FIDO2 hardware security keys** (WebAuthn, proof-of-possession, phishing-resistant, no secret on
  the wire, no platform-cloud trust), and it issues the opaque tokens. Machine callers obtain tokens
  the same way over the OAuth 2.1 flow. The MCP surface speaks the MCP authorization spec (protected
  resource metadata, `WWW-Authenticate` on 401) so an MCP client can complete the flow.
- **Per-tool authorization.** Each command declares a required **scope** (e.g. `pkg:vertical:read`,
  `pkg:vertical:write`); the caller's token must carry it. Default-deny: a command with no scope is
  unreachable, not wide-open. The write rail folds in — a write verb requires a `:write` scope.

## Closed by default (fail-closed)

The cardinal rule. A surface generated without an Authorization Server configured does not fall open
to anonymous access; it **refuses every call**. A no-tty surface cannot relocate confirmation outward
(the same reasoning as the write rail in GENERATION.md), so it fails safe and requires the capability
to be provisioned explicitly. Reads are not exempt: proprietary data is exactly what a read returns.

## Canon, enforced

Doctrine that is only professed drifts. The auth rail is enforced by
[`scripts/check_surface_auth.py`](scripts/check_surface_auth.py): a generated surface **fails the check**
when it ships without the auth gate, or when any exposed command lacks a scope. The check bites before
the rail is even built (a surface with no auth fails it), which is the point: it makes "closed by
default" a mechanical fact, not a hope. It runs in the deploy checks and is exercised by racecar's own
test suite.

## What this is not

- Not authentication of a *human at a browser page* (session login) bolted onto a machine API. The
  surfaces are machine interfaces; the human factor lives at the Authorization Server's login, which
  issues the token the machine then carries.
- Not mTLS on one surface and tokens on another. One path (AUTH-03). mTLS is a transport-layer add-on
  a deployer may stack at the edge, but it is device-bound and cannot be the surface's auth scheme,
  because MCP access must be device- and location-agnostic.
- Not a JWT. A self-describing token puts claims on the wire and resists revocation; the opaque
  handle does neither.
